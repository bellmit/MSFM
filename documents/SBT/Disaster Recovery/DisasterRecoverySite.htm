GUI Topics

Overview

The Disaster Recovery Site is a set of production systems that will be used if any disaster causes the current CBOE production system to become non-functional.  This guide is meant as a reference for SBT GUI developers who are familiar with the CBOE SBT Production system to learn the differences necessary to effectively support the DR site.

 

An overview of the systems comprising the SBT Disaster Recovery site can be found at:

 

http://ccserv1/documents/SBT/SBTTestEnvironments.html#dr

 

Notice that these systems’ names start with “dr.”  Whereas one production front end system is called prdfe01, a corresponding drfe01 exists at the disaster recovery site.  There is not, however, always a one-to-one correspondence for every production system, but there is for many of them.  The usernames and passwords used to support these systems are generally the same as they are for production, except for the GUI boxes.

 

The PC systems used to run various SBT GUI’s are named differently than they are in production and consist of three different groups:

§         14 Trade Ops systems named drto-21 through drto-37

§         4 System Ops systems named dropsmgmt-01, -02, -04,and –10

§         3 MDC Ops systems named drmdcops-01 and -06

 

Whenever we do a GUI rollout, now we must also include the appropriate Disaster Recovery PC’s in the rollout plan.  Just as QA performs the GUI installations for production, they will also install GUI’s on the Disaster Recovery PC’s.  However they will use a separate set of scripts, which exist because there are differences in the disaster recovery system configuration as compared to production.

Important differences between the Disaster Recovery site and our SBT production systems are:

§         location of firewalls and how client support tasks are affected: for example, currently there is a firewall between the 14 Trade Ops PC’s and their CAS systems; network support has opened certain ports in the firewall so that the GUIs can communicate with their CAS or SACAS.  If these ports do not stay open, for some reason, the GUI’s will not operate correctly.  See also How to FTP files to GUI PC’s and How to use REXEC with PC’s behind a firewall.

§         configuration changes and evolution: currently the DR site is in phase 1 of its evolutionary plan; new hardware and software, such as ICS, will be added; also, once enough PC’s are added for Trade Ops personnel, each help desk employee will have two pc’s at his/her desk, as they do now for production, and the GUI boxes will then be put behind the firewall where they belong.

WR number for DR is 3227.

GUI Rollout Scripts Differences Because of the DR System Configuration

The GUI rollout scripts are located in the ClearCase scripts vob under the GUIrollout directory:  Currently, there are three types of scripts: Production, IPD, and Disaster Recovery.

1)     REXEC is used three times per installation.  Each REXEC must use the -s switch because of the firewall.  The -s tells REXEC to use the same port for writing as for reading instead of trying to use two ports.

 

2)     Because the DR site hostnames are different from production, all references to hostnames in the GUI installation scripts are different. The differences are more extensive than just the first two or three letters of the hostname. The number of hostnames referenced varies, depending on the GUI, and ranges from only one (the CAS) in the Trader GUI to three or more in the IMM and SHM.

 

3)     At various times, the ftp strings used in the SAGUI and SHM were different.  Sometimes the FTP connection did not work with the %%2F and the complete file path. No explanation completely satisfied the observances of the problem, which may have been intermittent.

 

4)     The korn scripts copy a DOS script to the local machine.  Because of the above differences, the name of that DOS .bat script is different in each .ksh script

 

How to Log into GUI PC’s

There are 3 different ways to access the DR PC’s: two using devsup from a Unix system such as ccserv2, and a third that uses Remote Desktop connection software running on your PC. 

§         Log into devsup.  You need to have an account already set up on devsup and the RSA SecurID device given to you when you set up your account.  If you only have an account on devsupport, and not devsup, then use REXEC to accomplish what you need to do on the DR PC.  Then as soon as you can, request an account on devsup.

o       REXEC – lets you execute a program on a PC.  Often used to run “cmd” which gives you some of the same capabilities you have on a PC when using a DOS window.  Its advantage is that using REXEC takes only two more steps: sudo ssh to prdfe01 and the REXEC itself.

o       Use VNC on devsup.  If doing this from your PC, you will need an X-Windows software product, like Exceed, for the Linux devsup system to create windows on your PC (Hummingbird lets you try Exceed free for 30 days).  If, instead, the system you’re sitting in front of is Unix-based already, this may be your best choice.  You’ll have to set the DISPLAY variable to your X-Windows screen, then run VNC and supply the “drletmein” password to the PC.  (The password for DR-CONS-W2K is “letmeindr.”)

§         Use Remote Desktop connection software from your PC.  This multiple-step process will finally get you a window that acts just like you’re sitting in front of the DR site PC’s desktop.  You will have all the capabilities that the PC’s desktop gives you with decent responsiveness and performance.

How to use REXEC with PC’s behind a firewall

The port that REXEC uses, which is 512, should be opened in the firewall.  Contact Network support services to do this if it is not already open.

Use the -s switch in the REXEC command so that it will use the same port for writing as for reading instead of trying to use a second port.  For example:

rexec –s 172.28.72.1 cmd

How to use REXEC to copy files to a DR PC

rexec –s 172.28.72.1 C:/installation/elbow C:/to_PC_filename < from_local_file

How to FTP files to DR Site GUI PC’s

It is more difficult to get files over to the GUI boxes if they do not have REXEC running.  The non-intuitive, multiple-step process is:

 

1.      put the files onto deepfrzd from ccserv2 or own PC, etc.

2.      login to prdgc2a

cd tmp

ftp to deepfrz and get files

ftp to drgc02a and put files

3.      login to the gui pc you want files to go to

ftp to drgc02a and get files

 

The ftp step that crosses the firewall takes a significantly longer time to complete than the other ones.  Besides drgc02a, the only other system enabled for ftp access from the DR PC’s is drsacas01.

Reverse the above steps to copy files from a PC to ccserv2 or your own PC.

 

Troubleshooting

REXEC:

Cannot REXEC into PC System:

§         Can you ping the PC from prdfe01 (or wherever you’re running REXEC)?  Can you ping prdfe01 from the PC?

§         Is REXEC running as a service on the PC?  This cannot be verified by looking at the PC’s admin tool for looking at running services.  Instead:

Log into the PC

cd C:\installation  (if installation directory is not there, REXEC was not installed on the PC.  Get REXEC.)

Is Java (SDK or JRE) installed on the PC and its bin directory in the system path?  Test this by typing: java -version

Run install.bat (this installs REXEC)

Run startrexec.bat (this runs REXEC as a service) If you get: "java.net.BindException: Address already in use: JVM_Bind," then REXEC was already running as a service.

§         If you are trying to REXEC through a firewall, use the -s switch in the REXEC command so that it will use the same port for writing as for reading instead of trying to use a second port.  For example:

rexec –s 172.28.72.1 cmd

The PC asks for username and password:

§         The username and password that is sometimes requested by REXEC are not actually used for authentication.  If you’re running CBOE’s REXEC, this is not the reason why REXEC is not working.

The REXEC is still not working:

§         If REXEC is running, one last thing you can try is to telnet from prdfe01 to the REXEC port on the PC, which is 512.  You should see something like this if the REXEC process is running and reachable from prdfe01:

        Prdfe01 $ telnet dtro-01 512

        Trying 172.28.72.1...

        Connected to dtro-01.cboe.com.

        Escape character is '^]'.

If now you retry the REXEC, an error message may be printed that could give you a clue of why it isn’t working (for example, no space on disk, etc.)

§         You can also ask Network Support to use tcpdump or some listening program to help troubleshoot what is actually happening on the network when you try the REXEC.

Specifying the port number of the error port:

§         When production needs to set a fixed port for the error port used by REXEC, they use port number 1025:

rexec –e 1025 172.28.72.1 cmd

REXEC Help - Usage and Switches:

usage: rexec [-?] [-h ] [-n] [-p port] [-e] host command arg1 arg2 ...

purpose: to execute a command on a remote host

switches:

-? or -h  = print out these instructions and exit.

-n = ask the user for a user name and a password. The default action is to retrieve these from .netrc if possible.

-p port = specify the port number of the server. The default is to look up the service by name.

-e port = use the specified port for the stderr connection. This is useful for getting past firewalls, while keeping the separate channels.

-s = use a single connection for both stdout and stderr. This is useful for getting past firewalls.

General Topics

devsup Instructions

devsup is a Linux box that has been configured to allow SBT developers to access the SBT servers.  The production SBT servers must be accessed only after you've notified Operations or you have been called by Operations.

Once your account is enabled and you have an RSA SecurID device:

§         SSH to devsup (170.137.248.89).  For an SSH client, you can either use a development UNIX box or Putty, an excellent SSH client for Windows (and free).  Access to devsup is given only with your CBOE username, so use the –l username switch with SSH, if necessary, to specify your username if you are not currently logged in with your own username.

§         If devsup asks for your username, then supply it.

§         For password, enter your SecurID passcode (your devsup PIN + 6 digits on RSA SecurID)

§         You will now be able to use ssh (for shell access) or sftp (for file transfers) to access the production SBT servers through sudo

Shell access example: signing on to prdfe01 as the infrap user

                $ sudo ssh -l infrap prdfe01

File transfer example: signing on to prdbc01b as the infrap user to transfer files

                $ sudo sftp infrap@prdbc01b

Please note that an e-mail will be sent to Operations management whenever you invoke sudo.

Remote Desktop Client Instructions

Remote desktop connection software provides a fast, secure means of remotely accessing the DR site PC’s.  If you happen to have Windows XP on your PC, you will already have the Remote Desktop client software on your machine.  Otherwise, read the instructions and download the client software from:

http://intranet/departments/InformationSecurity/RemoteAccess.html

To use Remote Desktop and host mstsdv.cboe.com to access the DR site:

1.      Start the Remote Desktop connection client on your PC.  It can often be found from the start menu at:

Start ▬► Programs ▬► Accessories ▬► Communications ▬► Remote Desktop Connection

The “Remote Desktop Connection” window should pop up.  If you don’t see it, you may have to click on the new window’s icon if the window is occluded.

Enter mstsdv.cboe.com

If you do not see a General tab, click on the Options>> button.

On the General tab, enter your workstation ID (your ID on your CBOE PC) in the User name field. 

Enter your PC’s domain password.

Check the domain, and if necessary change it to the domain you use to login to your CBOE PC. (Many people are on ICLAN, but others are on DSDL.CORP.CBOE).

Click on Connect.

2.      On "mstsdv.cboe.com - Remote Desktop” window that pops up, click OK on the Warning window that asserts that system use is only for authorized business purposes.

Log into the Novell Client window that pops up:  

§               Click on the Advanced>> button to check, and if necessary correct, the Context (the same domain entered above). 

§               Enter your Username and Password.

§               Click OK.

Scroll down until you see the start menu, if necessary, and select:

Start ▬► Programs ▬► VNC Viewer ▬► VNC Viewer

3.      In the Connection window, enter the IP address of the DR PC you want.

4.      In the “VNC Authentication” window, enter password: drletmein and click Log On.  (The password for DR-CONS-W2K is letmeindr.)

5.      In the Novell login window, you will login to the PC as “Workstation Only,” i.e. local context to the DR PC.

NOTE:  For some reason, at this point, you have to login successfully the first time you try or you will never get past this point.  So, if you are not able to login successfully the first time you try, go back to the mstsdv window, and re-run the VNC Viewer.

Carefully enter the local login for DR PC.  Most of them have username and password: ops / ops.  Click on the OK button.

If the login works, you will be looking at the desktop screen of the PC, which should at least have the toolbar with the Start menu on it, and may have desktop icons and windows of other running programs as well.  If you only have a completely blank window before you, go back to the mstsdv window, and re-run the VNC Viewer.
